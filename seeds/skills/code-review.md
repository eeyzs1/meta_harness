# Code Review (Phase-Aware)

吸收自：union 框架 `skills/code-review/SKILL.md`
默认 Phase：TEST（以 `project.yaml` → `phase_skills` 为准）

## 核心原则

**每个变更在发布前都经过审查。审查关注正确性，不是风格偏好。**

## 审查清单

### Spec 合规（第一遍）
- 代码实现了 spec 中的一切吗？
- 是否添加了 spec 中没有的东西？（过度构建和构建不足一样糟糕）
- 所有验收标准都处理了吗？

### 代码质量（第二遍）
- **正确性**：边界情况已处理，错误状态已覆盖
- **模式**：遵循现有代码库约定
- **性能**：没有明显的 N+1 查询，没有不必要的操作
- **安全**：输入验证，代码中没有密钥，没有注入面
- **测试**：测试存在，测试真实行为，覆盖边界情况

### 交付物检查
- Phase spec 中列出的所有文件都存在
- 没有意外的文件创建
- 没有在声明范围外修改文件

## 审查结论

- **PASS** — 所有标准满足，没有问题
- **PASS_WITH_NOTES** — 小的观察，没有阻塞性问题
- **NEEDS_FIX** — 特定问题必须在继续前解决
- **BLOCKED** — 根本性问题，需要重新规划

## 反模式

| 不要 | 改为 |
|---|---|
| 挑剔格式 | 让 linter 处理 |
| 审查而不读 spec | 始终先对照 spec 审查 |
| 没有证据的"LGTM" | 说明你检查了什么和发现了什么 |
| 因为"简单"跳过审查 | 简单代码也有 bug |

## 接收审查反馈

### 核心原则
**验证后再实施，询问后再假设。** 技术正确性优于社交舒适。

### 响应模式
```
收到审查反馈时:
1. 阅读 — 完整阅读反馈，不急于反应
2. 理解 — 用自己的话复述，或提问澄清
3. 验证 — 对照代码库实际情况检查
4. 评估 — 对该代码库是否技术正确？
5. 响应 — 技术确认或有理有据的推回
6. 实施 — 逐条处理，每项测试
```

### 禁止的回应
- "你说得对！" / "好主意！"（表演性赞同）
- "让我立即实施"（验证之前）
- 任何感谢表达（用行动代替言语）

### 何时推回
- 建议会破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对该技术栈技术不正确
- 与架构决策冲突

推回方式：使用技术推理，而非防御性态度。

### 实施顺序
多条目反馈时：先澄清所有不清楚的 → 阻塞性问题 → 简单修复 → 复杂修复 → 逐项测试。

## 在 Meta-Harness 中的上下文

- 在 TEST Phase 自动加载
- 审查结论记录为 `REVIEW_VERDICT` 事件
- 审查中发现的问题触发修复循环（implementer 修复 → 重新审查）
- 最终审查是审计的一部分：对照原始 ROADMAP 重新验证
