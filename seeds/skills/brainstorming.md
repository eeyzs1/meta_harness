# Brainstorming (Phase-Aware)

吸收自：union 框架 `skills/brainstorming/SKILL.md`
默认 Phase：PLAN（以 `project.yaml` → `phase_skills` 为准）

## 核心原则

**Design before code. Always.** 即使 5 分钟的设计也能节省数小时的错误实现。

<HARD-GATE>
在呈现并通过设计之前，不要编写任何代码、搭建任何项目、或调用任何实现技能。
</HARD-GATE>

## 何时激活

- 在任何 Meta-Harness 运行的 **PLAN** Phase 自动加载
- 可手动触发："brainstorm X", "design Y", "让我们想想 Z"

## 流程

### 1. 探索项目上下文
- 读取 `project.yaml` 获取仓库结构、技术栈、约定
- 阅读与任务相关的现有代码和文档
- 识别要遵循的现有模式

### 2. 提出澄清问题（一次一个）
- 理解：目的、约束、成功标准
- 尽可能使用多选题
- 每条消息只问一个问题
- 不要问代码库已经回答的问题

### 3. 提出 2-3 种方案
- 每种方案说明权衡
- 首先推荐你的首选方案及理由
- 考虑：简洁性、可维护性、与现有模式的一致性

### 4. 呈现设计
- 每个部分根据复杂度调整篇幅（几句话到 200-300 字）
- 覆盖：架构、组件、数据流、错误处理、测试
- 每部分呈现后询问是否看起来正确
- 对现有代码库：遵循现有模式，包含有针对性的改进

### 5. 将设计写入 Spec
- 保存到 `.meta-harness/runs/<id>/phases/phase-1/spec.md`
- 运行自我审查：检查占位符、矛盾、歧义、范围

## 反模式："这太简单了"

每个项目都要经过这个流程。即使是单函数工具。"简单"项目是未检查假设造成最多浪费的地方。

## 红线

| 想法 | 现实 |
|---|---|
| "这太简单了，不需要设计" | 简单的事情隐藏着假设 |
| "我边做边想" | 这是构建错误东西的方式 |
| "设计里再加一个功能" | YAGNI。删掉它。 |
| "用户说'构建 X'，所以我知道他们想要什么" | 你知道他们说了什么，不知道他们需要什么 |

## 与 Meta-Harness 的连接

此技能直接输入引擎的规划流水线：
- 输出成为 `phase-1/spec.md`（PLAN Phase spec）
- 设计决策流入 `ROADMAP.md`
- 风险分析流入 `THINKING.md`
