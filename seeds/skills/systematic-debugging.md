# Systematic Debugging (Phase-Aware)

吸收自：union 框架 `skills/systematic-debugging/SKILL.md`
默认 Phase：TEST（失败时触发）或任何遇到错误的 Phase（以 `project.yaml` → `phase_skills` 为准）

## 核心原则

**始终先找到根因再尝试修复。症状修复就是失败。**

<IRON-LAW>
没有根因调查之前，不做任何修复
</IRON-LAW>

## 四个 Phase

### Phase 1: 根因调查
1. **仔细阅读错误消息** — 它们通常包含精确的解决方案
2. **一致复现** — 精确步骤，每次一致
3. **检查最近的变更** — git diff，最近的提交，配置变更
4. **追踪数据流** — 坏值从哪里来？反向追踪调用栈
5. **在多组件系统中**：在每个边界添加诊断工具，收集证据，识别失败组件

### Phase 2: 模式分析
1. 在同一代码库中找到工作的例子
2. 与参考实现比较
3. 识别每个差异，无论多小
4. 理解依赖和假设

### Phase 3: 假设和测试
1. 形成一个假设："我认为 X 是根因，因为 Y"
2. 最小化测试 — 一次一个变量
3. 验证后再继续
4. 如果错了，形成新的假设。不要堆砌修复。

### Phase 4: 实施
1. 创建复现 bug 的失败测试用例
2. 实施针对根因的修复
3. 验证：测试通过，没有其他测试被破坏，问题已解决
4. 如果 3+ 次修复都失败了 → **质疑架构**，而不是修复

## 红线 — STOP

如果你在想："快速修复一下"，"试试改 X"，"跳过测试"，"再试一次修复" — STOP。回到 Phase 1。

## 条件等待（Condition-Based Waiting）

**绝不使用任意超时或 sleep。** 使用基于事件的等待：
- 用轮询条件（如文件存在、进程退出）替代 `sleep(1000)`
- 设置最大等待时间（如 30s）作为安全网，不是主要机制
- 条件满足后立即继续，不要等待完整超时

## 防御深度（Defense-in-Depth）

对于多组件 bug：
1. 在每个组件边界添加诊断日志
2. 收集跨所有组件的证据
3. 识别失败起源于哪个组件
4. 将调试范围缩小到该组件
5. 不要跨多个组件同时修复

## 根因追踪（Root Cause Tracing）

```
症状 → 直接原因 → 中间原因 → 根因

示例：
  症状：API 返回 500
  直接原因：数据库查询失败
  中间原因：连接池耗尽
  根因：连接泄漏在上一个部署中未修复
```

## 在 Meta-Harness 中的上下文

- 当 Phase 的 VERIFY 块显示失败时自动加载
- 3-strike 恢复协议：第 1 次失败 → 自动重试加 probe；第 2 次 → 写 fix-spec；第 3 次 → HANDOFF
- 调试证据记录在 `phase-N/evidence/`
- 根因发现在升级时写入 `phase-N.fix.md`
