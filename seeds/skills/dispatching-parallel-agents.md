# Dispatching Parallel Agents (Cross-Phase)

吸收自：union 框架 `skills/dispatching-parallel-agents/SKILL.md`
触发：当存在多个独立故障或调查任务时（跨 Phase）

## 核心原则

**每个独立问题域派发一个代理。让它们并发工作。**

你将任务委派给具有隔离上下文的专门代理。通过精确构建它们的指令和上下文，确保它们聚焦并成功完成任务。它们绝不应该继承你会话的上下文或历史——你构建它们确切需要的东西。

当你有多个不相关的故障（不同测试文件、不同子系统、不同 bug）时，顺序调查浪费时间。每个调查是独立的，可以并行进行。

## 何时使用

```
多个故障？ → 是 → 它们独立吗？ → 是 → 可以并行工作吗？ → 是 → 并行派发
              ↓                    ↓                        ↓
             否                   否                        否
              ↓                    ↓                        ↓
         单一代理调查          相关，一起调查              顺序代理
```

**使用场景：**
- 3+ 个测试文件因不同根因失败
- 多个子系统独立损坏
- 每个问题无需其他上下文即可理解
- 调查之间无共享状态

**不使用场景：**
- 故障相关（修复一个可能修复其他）
- 需要理解完整系统状态
- 代理会互相干扰

## 模式

### 1. 识别独立领域

按损坏的内容分组故障：
- 文件 A 测试：工具审批流程
- 文件 B 测试：批量完成行为
- 文件 C 测试：中止功能

每个领域是独立的——修复工具审批不影响中止测试。

### 2. 创建聚焦的代理任务

每个代理获得：
- **特定范围：** 一个测试文件或子系统
- **明确目标：** 使这些测试通过
- **约束：** 不要更改其他代码
- **期望输出：** 你发现和修复了什么摘要

### 3. 并行派发

```
Agent 1 → Fix agent-tool-abort.test.ts
Agent 2 → Fix batch-completion-behavior.test.ts
Agent 3 → Fix tool-approval-race-conditions.test.ts
```

### 4. 审查和集成

代理返回后：
- 阅读每个摘要
- 验证修复不冲突
- 运行完整测试套件
- 集成所有变更

## 代理 Prompt 结构

好的代理 prompt：
1. **聚焦** — 一个清晰的问题域
2. **自包含** — 理解问题所需的所有上下文
3. **输出明确** — 代理应该返回什么？

## 常见错误

**❌ 太宽泛：** "Fix all the tests" — 代理迷失
**✅ 具体：** "Fix agent-tool-abort.test.ts" — 聚焦范围

**❌ 无上下文：** "Fix the race condition" — 代理不知道在哪里
**✅ 上下文：** 粘贴错误消息和测试名称

**❌ 无约束：** 代理可能重构一切
**✅ 约束：** "Do NOT change production code" 或 "Fix tests only"

**❌ 模糊输出：** "Fix it" — 你不知道什么变了
**✅ 具体：** "Return summary of root cause and changes"

## 验证

代理返回后：
1. **审查每个摘要** — 理解什么变了
2. **检查冲突** — 代理编辑了相同代码吗？
3. **运行完整套件** — 验证所有修复一起工作
4. **抽查** — 代理可能犯系统性错误

## 在 Meta-Harness 中的上下文

- 此技能**跨 Phase** 操作：可在 PLAN（多个设计问题）、IMPLEMENT（多个独立任务卡片）或 TEST（多个独立测试故障）期间触发
- 与 `subagent-driven-dev` **不同**，后者用于单个 IMPLEMENT Phase 中计划任务列表的顺序实现
- 引擎的审计日志将每次派发记录为单独事件，支持运行后并行效率分析
- 在 TEST Phase 中使用时，此技能与 `systematic-debugging` 自然配对——每个代理可独立应用该技能
- leaf-protocol（`task.json` / `result.json`）提供代理输入/输出的标准化接口
- Worktree 隔离至关重要：每个派发的代理应在自己的 worktree 中操作以避免文件冲突
